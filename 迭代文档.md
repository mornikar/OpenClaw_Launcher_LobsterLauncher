### V1.1 - 修复右键删除本地服务功能

**日期**: 2026-05-15  
**问题**: 点击右键删除本地服务时，显示"删除服务失败：Error: An object could not be cloned"

---

#### 问题描述

用户在右键菜单中点击"🗑 删除"按钮时，Electron抛出序列化错误：
```
Error: An object could not be cloned
```

**触发路径**:
1. 右键点击本地服务卡片
2. 选择"🗑 删除"菜单项
3. 显示确认对话框
4. 点击"确定"按钮
5. ❌ 报错：删除服务失败：Error: An object could not be cloned

---

#### 问题根因分析

**核心原因**: Vue响应式对象无法通过Electron IPC传递

**技术细节**:

1. **Vue响应式系统**:
   - `localServices`使用`ref()`创建，内部包含Proxy包装
   - `confirmDialog.value`是Vue响应式对象
   - `confirmDialog.value.onConfirm`闭包引用了响应式变量

2. **Electron IPC序列化限制**:
   - Electron的`ipcRenderer.invoke()`使用结构化克隆算法
   - 无法序列化Proxy对象
   - 无法序列化包含Proxy引用的闭包函数

3. **错误链路**:
   ```
   deleteLocalService(id)
   → confirmDialog.value.onConfirm = async () => { ... }  // 闭包引用响应式对象
   → confirmDialog.value.show = true  // 整个对象被存储
   → 用户点击确定
   → onConfirm()执行
   → window.api.saveServices(servicesList)  // 尝试序列化localServices.value
   → ❌ localServices.value包含Proxy包装的响应式对象
   → IPC序列化失败：An object could not be cloned
   ```

4. **之前的修复尝试（失败）**:
   - 使用全局变量`pendingDeleteServiceId`避免闭包引用
   - 使用`JSON.parse(JSON.stringify())`深度克隆服务列表
   - **问题**: `confirmDialog.value`本身是Vue响应式对象，在赋值时已经被Proxy包装
   - 即使闭包内使用纯对象，整个`confirmDialog.value`对象在渲染时仍会被尝试序列化

---

#### 解决方案

**架构调整**: 将删除逻辑移到主进程，避免渲染进程传递Vue响应式对象

**实现思路**:
1. 渲染进程只传递纯字符串（服务ID）给主进程
2. 主进程直接操作服务配置文件
3. 渲染进程接收成功/失败结果后更新本地状态

**优势**:
- ✅ 完全避免Vue响应式对象序列化问题
- ✅ 最小化IPC通信数据量
- ✅ 主进程直接操作文件，更安全
- ✅ 代码更简洁，职责分离清晰

##### 步骤1: 修改渲染进程 (HomePage.vue)

**文件**: `src/renderer/src/components/HomePage.vue`

**修改2**: 重写`deleteLocalService`函数（第813-860行）
```javascript
// 删除本地服务
async function deleteLocalService(id: string) {
  try {
    // 先找到要删除的服务,用于确认对话框
    const service = localServices.value.find(s => s.id === id)
    if (!service) {
      console.error('服务不存在:', id)
      return
    }
    
    // 存储待删除的服务信息到全局变量(避免闭包序列化问题)
    pendingDeleteServiceId = id
    pendingDeleteServiceName = service.name
    
    // 显示确认对话框
    confirmDialog.value = {
      show: true,
      title: '删除本地服务',
      message: `确定要删除服务"${service.name}"吗?\n\n注意:这只会从启动器中移除该服务,不会删除文件。`,
      onConfirm: async () => {
        confirmDialog.value.show = false
        try {
          const deleteId = pendingDeleteServiceId
          const deleteName = pendingDeleteServiceName
          if (!deleteId) {
            console.error('服务ID不存在')
            return
          }
          
          // 调用主进程的删除服务方法,避免Vue响应式对象序列化问题
          const result = await window.api.deleteService(deleteId)
          if (result.success) {
            // 更新本地状态,使用JSON深度克隆避免IPC序列化错误
            localServices.value = JSON.parse(JSON.stringify(
              localServices.value.filter(s => s.id !== deleteId)
            ))
            console.log('✅ 服务删除成功:', deleteId)
          } else {
            console.error('❌ 删除服务失败:', result.error)
            alert('删除服务失败:' + (result.error || '未知错误'))
          }
        } catch (e) {
          console.error('❌ 删除服务异常:', e)
          alert('删除服务失败:' + String(e))
        } finally {
          pendingDeleteServiceId = null
          pendingDeleteServiceName = null
        }
      }
    }
  } catch (e) {
    console.error('删除服务时出错:', e)
  }
}
```

**关键变化**:
- ~~之前~~: 在`onConfirm`中直接过滤`localServices.value`并调用`window.api.saveServices()`
- ~~现在~~: 在`onConfirm`中调用`window.api.deleteService(deleteId)`，只传递服务ID字符串

    const services = loadServicesConfig()
    const updatedServices = services.filter(s => s.id !== serviceId)
    saveServicesConfig(updatedServices)
    return { success: true }
  } catch (e) {
    log.error('删除服务失败:', e)
    return { success: false, error: String(e) }

})

#### 技术要点总结

**Electron IPC序列化规则**:
1. 只能传递可序列化的数据（字符串、数字、布尔值、数组、对象）
2. 无法传递Proxy、Function、DOM节点等
3. Vue的`ref()`和`reactive()`创建的响应式对象包含Proxy包装，不能直接传递

**避免序列化错误的最佳实践**:
1. **最小化IPC通信**: 只传递必要的纯数据（字符串、数字、JSON对象）
2. **使用JSON深度克隆**: `JSON.parse(JSON.stringify(obj))`移除Proxy包装
3. **主进程操作数据**: 需要修改配置时，让主进程直接操作文件
4. **避免闭包引用**: 不要在IPC回调中使用Vue响应式变量

**Vue响应式系统**:
- `ref()`创建的对象被Proxy包装
- `localServices.value`返回的是Proxy包装的数组
- `localServices.value.filter()`返回的数组元素仍然是响应式对象
